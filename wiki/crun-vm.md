# CRUN-VM — OCI Runtime for Running VMs as Containers

| Field | Value |
|---|---|
| **Origin** | [containers/crun-vm](https://github.com/containers/crun-vm) |
| **License** | Apache 2.0 |
| **Stack** | Rust, QEMU, libvirt, crun |
| **Source** | `sources/crun-vm/` |
| **Wanted** | Run QEMU VMs as OCI containers — manage VMs with the same tooling as containers |

## What it is

crun-vm is a thin OCI runtime shim that sits between the container engine (Podman/Docker/CRI-O) and the real OCI runtime `crun`. It intercepts container creation and rewrites the OCI config.json to set up a QEMU/KVM virtual machine instead of running the container process directly. The container becomes the VM host.

**Key insight:** It's NOT a full OCI runtime — it's a shim that delegates most lifecycle operations to `crun`. The "container" provides the environment for QEMU to run inside, with the VM image as its payload.

## Architecture

### Lifecycle Mapping

```
podman run --runtime crun-vm quay.io/containerdisks/fedora:40
```

| OCI Operation | crun-vm Behavior |
|---|---|
| **create** | The main work: loads OCI config.json, detects container engine, creates a private directory, generates QEMU domain XML, sets up cloud-init/Ignition config, sets up virtiofs mounts, generates SSH keys, delegates to `crun create` |
| **start** | Delegated 100% to `crun` |
| **stop/kill** | SIGTERM → entrypoint.sh traps it → `virsh shutdown domain` for graceful QEMU shutdown |
| **exec** | Intercepts exec process config, rewrites to run `exec.sh` which SSHes into the VM, delegates to `crun exec` |
| **delete** | Wraps `crun delete` — calls `crun delete` internally, then cleans up crun-vm mounts (.ssh, image overlay) |
| **run/checkpoint/events/pause/resume** | All rejected — not implemented |

### QEMU Inside the Container

The container entrypoint (PID 1) runs `entrypoint.sh` which:
1. Starts `virtlogd`, then `libvirtd` (or `virtqemud`)
2. Defines the libvirt domain from generated `domain.xml`
3. Starts the VM with `virsh start domain --console`
4. Traps SIGTERM → `virsh shutdown domain` → waits for VM power-off

### Libvirt/QEMU Domain Configuration

Generated by `src/commands/create/domain.rs`:

- **Domain type:** `kvm` (default) or `qemu` (emulated, if `--emulated`)
- **Firmware:** EFI with secure-boot disabled
- **CPU:** host-passthrough mode
- **Machine type:** q35 (x86/x86_64)
- **vCPUs:** From OCI CPU quota, or host CPU count
- **Memory:** From OCI memory limit, default 2 GiB
- **ACPI:** Enabled (for graceful shutdown via `virsh shutdown`)
- **Main disk:** virtio-blk, qemu driver, qcow2 format
- **Additional block devices:** For `--device` and `--blockdev` passthrough
- **Cloud-init ISO:** Separate virtio-blk disk (not passthrough)
- **Network:** user-mode with passt backend, virtio NIC
- **Serial console:** PTY type
- **virtiofs:** Each mounted dir gets a virtiofs filesystem

### Networking

```
Host → Container netns → passt → QEMU virtio-net → VM
```

- Container gets a standard OCI network namespace (managed by Podman/Docker)
- passt runs inside the container, translating between the container's netns and the VM's virtio-net
- Port forwarding: Podman's `-p` flag creates iptables rules → passt forwards into the VM
- crun-vm adds CAP_SYS_CHROOT and relaxes seccomp for passt
- No host networking mode, no macvlan/ipvlan passthrough

### virtiofs (File Sharing)

- Each mounted directory gets a virtiofs filesystem
- Uses `/usr/libexec/virtiofsd` inside the container (via wrapper script `embed/virtiofsd.sh`)
- Sandbox mode: chroot
- Directory mounts from the original OCI spec become virtiofs mounts

## Bootc Compatibility

crun-vm has **first-class bootc support**. When it detects a bootc container (`/usr/lib/bootc/install`), it:

1. Launches `embed/bootc/prepare.sh` asynchronously during `create`
2. This script generates a VM disk image using krun (the VM disk builder) from the bootc container image
3. The generated disk becomes the VM's main storage
4. On `--persistent` mode, the overlay is writable (writes propagate to the original file)

Restrictions:
- bootc containers require Podman or Docker (not bare CRI-O/Kubernetes)
- Incompatible with `--emulated` mode (needs KVM)
- Requires KVM passthrough on the host

## Requirements

- **Linux host** with `/dev/kvm` (KVM acceleration)
- **No macOS support** — QEMU on macOS lacks KVM acceleration; requires HVF on Apple Silicon but crun-vm uses KVM-specific paths
- **No budget VPS support** — most VPS do not expose `/dev/kvm` to guests
- **Nested virtualization required** — needs a host that provides KVM passthrough
- QEMU, libvirt, passt installed on the container host (the crun-vm binary is pulled into the container)

## Limitations

- No support for checkpoint/restore, pause/resume, or `podman run` (the `run`, `checkpoint`, `events`, `pause`, `ps`, `resume`, `update` commands are all rejected)
- SSH key management: host user's key is used if running with rootless Podman in the host user namespace; otherwise a random key is generated
- No host networking mode
- Container startup is significantly slower than bare containers (VM boot time: 5-30s vs 100ms)
- Memory overhead: guest kernel + QEMU + libvirt adds ~1-2 GB RAM
- seccomp relaxation for passt is a noted TODO (not ideal security practice)

## Deployment Role

crun-vm fills a specific niche: **deepest isolation on bare metal**. For production use:

- **On dedicated servers with KVM passthrough:** crun-vm provides the strongest isolation boundary — a full hardware VM around each agent stack
- **On macOS/Linux developer machines:** Useful for testing bootc images locally with `podman run --runtime crun-vm`
- **On budget VPS (no nested virt):** NOT available — fall back to rootless Podman alone
